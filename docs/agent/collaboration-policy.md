# AI-Human Collaboration Policy

AI의 과도한 코딩을 방지하고, 사람의 판단과 학습을 중심에 두는 협업 정책입니다.

## 핵심 철학

> "AI는 생각을 돕고, 사람은 결정하고 검증한다."

AI가 모든 코드를 작성하면:
- 개발자의 코드 이해도가 낮아짐
- 버그 발생 시 디버깅 어려움
- 테스트의 본질적 가치(요구사항 검증) 상실

## 에이전트 팀 구성

| 에이전트 | 역할 | 정의 파일 |
|---------|------|----------|
| **Claude Code (기본)** | 코드베이스 탐색, 구현, 테스트 실행, 리팩토링 제안 | - |
| **reviewer** | 요구사항 검토, 개발 내용 리뷰, 테스트 결과 분석, 리뷰 MD 작성 | `.claude/agents/reviewer.md` |

> 리뷰 문서 출력 경로: `docs/reviews/{도메인}-review-{날짜}.md`

## 테스트 코드 협업 워크플로우

```
1. [AI] 기능 요구사항 분석
        ↓
2. [AI] 테스트 시나리오 목록 제안
        ↓
3. [Human] 시나리오 검토/수정/승인
        ↓
4. [AI] 테스트 템플릿 제공 (선택적)
        ↓
5. [Human] 테스트 코드 직접 작성  ← 핵심
        ↓
6. [AI] 테스트 실행 및 결과 보고
        ↓
7. [Reviewer] 테스트 결과 분석 → docs/reviews/ 에 리뷰 MD 작성
        ↓
8. [Human] 실패 시 수정 또는 AI에게 구현 코드 수정 지시
```

## 개발 완료 후 리뷰 워크플로우

```
1. [AI] 구현 완료
        ↓
2. [Reviewer] plan.md ↔ 구현 코드 비교 검토
        ↓
3. [Reviewer] 컨벤션 준수 여부 확인
        ↓
4. [Reviewer] 테스트 실행 및 결과 분석
        ↓
5. [Reviewer] docs/reviews/{도메인}-review-{날짜}.md 작성
        ↓
6. [Human] 리뷰 문서 확인 후 머지 판단
```

## AI의 테스트 관련 역할

### 할 수 있는 것
- 테스트 시나리오 목록 제안
- Given-When-Then 구조 템플릿 제공
- 테스트 실행 및 결과 보고
- 실패 원인 분석 및 수정 방향 제안
- 기존 테스트 코드 분석/설명

### 하지 않는 것
- 새 테스트 코드 직접 작성
- 테스트 로직 구현
- 테스트 데이터 생성 코드 작성

## 예시: 테스트 시나리오 제안

AI가 제공하는 형태:

```markdown
## MemberService.createMember 테스트 시나리오

### 정상 케이스
1. 유효한 회원 정보로 생성 시 → 회원 정보 반환
2. 중복되지 않은 userId로 생성 시 → 성공

### 예외 케이스
3. 이미 존재하는 userId로 생성 시 → DUPLICATE_USER_ID 예외
4. 이미 존재하는 email로 생성 시 → DUPLICATE_EMAIL 예외
5. 필수 필드 누락 시 → INVALID_INPUT 예외

### Edge 케이스
6. 최대 길이 userId로 생성 시 → 성공
7. 특수문자 포함 name으로 생성 시 → 성공
```

Human이 이를 바탕으로 실제 테스트 코드를 작성합니다.

## 이 정책의 이점

1. **학습 효과**: 테스트 작성을 통해 요구사항을 깊이 이해
2. **품질 향상**: 사람이 직접 검증 로직을 고민
3. **디버깅 용이**: 테스트 코드를 이해하고 있으므로 문제 해결 용이
4. **AI 의존성 감소**: 핵심 검증 로직은 사람이 통제

## 예외 상황

다음 경우에는 AI가 테스트 코드를 작성할 수 있음:
- 명시적으로 요청한 경우
- 기존 테스트 코드의 단순 복제/수정
- 보일러플레이트 코드 생성
